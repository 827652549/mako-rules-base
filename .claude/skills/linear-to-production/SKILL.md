# linear-to-production

## 用途

将 Linear 任务从 PRD 阶段推进到生产上线，覆盖设计、实施、审查、部署的完整流程。

## 触发方式

用户输入 `/linear-to-production MAK-XXX`，其中 `MAK-XXX` 为 Linear 任务编号。

## 执行流程

### 阶段 1：读取任务 & 设计方案

1. 调用 Linear MCP `get_issue` 读取任务详情（标题、描述、状态等）
2. **强制进入 Plan 模式**（使用 `EnterPlanMode`），基于任务需求、现有代码结构和项目规范设计方案
3. **调用 `ExitPlanMode` 等待用户确认方案**。用户在确认前可能会提出修改意见，根据反馈调整方案后再次等待确认
5. 用户确认后方可进入实施阶段

### 阶段 2：实施

1. 按照已确认的方案逐步实施代码变更
2. 实施过程中**无需再次确认**，直接推进到完成
3. 如果涉及数据库 schema 变更（Drizzle schema 修改），自动生成模拟数据脚本或 SQL 语句，供用户后续验证
4. 保持变更最小化，仅修改与当前任务相关的代码

### 阶段 3：质量检查

1. **代码规范审查**：检查所有新增/修改的代码，确保：
   - 无硬编码的临时数据或调试代码
   - 遵循项目现有的代码风格和目录结构
   - TypeScript 类型正确，无 `any` 滥用
   - 如果发现不符合的项，尝试自动修复

### 阶段 4：提交和审查

调用 `/提交和审查` skill：`Skill(skill="提交和审查")`

### 阶段 5：更新 Linear

1. 使用 Linear MCP `save_issue` 将任务状态更新为 `Done`
2. 使用 Linear MCP `save_comment` 在任务中添加评论，格式：

```
✅ 已完成并部署到 Vercel 测试环境。

## 实施总结

[用 2-3 句话概括本次做了什么、解决了什么问题]

### 变更文件
- `path/to/file.ts` — [做了什么改动]
- `path/to/another.ts` — [做了什么改动]

### 技术要点
- [关键实现细节或设计决策]
- [值得注意的地方]

🤖 由 [模型名称] 生成
```

评论内容要求：
- **实施总结**：简明扼要说明做了什么、为什么做，不超过 3 句话
- **变更文件**：列出主要变更文件及一句话说明，不超过 5 个文件
- **技术要点**：关键实现细节，有则写，无则省略此节
- 模型名称从当前会话的模型标识中获取

## 注意事项

- 每个阶段遇到网络错误时自动重试（等待 10 秒后重试，最多 3 次），不要求用户手动介入
- 执行报告由 `/提交和审查` 子流程自动生成
- 全程使用中文与用户交互
