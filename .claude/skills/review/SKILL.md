# Review — 结构化代码审查

你正在执行代码审查任务。只审查，不修改任何代码。

## 流程

### 1. 收集变更

```bash
git diff release...HEAD --stat
git diff release...HEAD
```

- 列出所有变更文件
- 统计新增/删除行数
- 如无变更，报告"无待审查代码"并结束

### 2. 逐文件审查

对每个变更文件，按以下 6 个维度检查：

| 维度 | 检查内容 | 权重 |
|------|----------|------|
| 🔴 类型安全 | `any` 类型使用、类型断言滥用、缺少类型注解 | 高 |
| 🔴 运行时风险 | 空值未处理、边界条件遗漏、未捕获异常 | 高 |
| 🟡 代码质量 | 重复代码、过长函数、命名不清晰、魔法数字 | 中 |
| 🟡 架构适配 | 是否符合项目三层架构、是否跨层直接调用 | 中 |
| 🔵 性能 | 不必要的重渲染、缺失 memo/useCallback、大数据量无分页 | 低 |
| 🔵 规范合规 | 是否通过 lint、编码风格是否一致、注释是否充分 | 低 |

### 3. 输出审查报告

用以下格式输出：

```
## 📋 代码审查报告

### 概览
- 变更文件: {N} 个
- 新增: +{lines} / 删除: -{lines}
- 总体评级: 🟢 通过 / 🟡 需修改 / 🔴 不可合并

### 发现

#### 🔴 必须修复（Critical）
1. **{文件名}:{行号}** — {问题描述}
   - 影响: {可能导致的问题}
   - 建议: {修复方案}

#### 🟡 建议修改（Warning）
1. ...

#### 🔵 可选优化（Info）
1. ...

### 各维度评分
| 维度 | 评分 | 说明 |
|------|------|------|
| 类型安全 | ✅/⚠️/❌ | ... |
| 运行时风险 | ✅/⚠️/❌ | ... |
| 代码质量 | ✅/⚠️/❌ | ... |
| 架构适配 | ✅/⚠️/❌ | ... |
| 性能 | ✅/⚠️/❌ | ... |
| 规范合规 | ✅/⚠️/❌ | ... |

### 结论
{一句话总结，给出明确的合并建议}
```

## 约束

- 只读不改，不修改任何文件
- 指出问题时必须给出具体文件路径和行号
- 不输出空洞的"建议优化"，每条发现必须有实际影响说明
- 如果代码质量良好，明确说"建议合并"，不要硬找问题
